# Selective training artifact provenance

Training reuses a named Configuração de Modelo only when its manifest proves that its configuration, resolved training recipe, relevant dependency versions, Divisão Estratificada, consumed inputs, and essential output hashes still match. Freshness is evaluated per configuration: adding a configuration or implementation trains only its users, changing model-specific knowledge invalidates only its users, and changing shared training knowledge such as `train.py` invalidates every configuration. Execution backend is recorded for provenance but CPU versus CUDA does not invalidate an otherwise current evaluation.

Each recipe identifies the specific implementations and external dependencies it uses instead of hashing whole multi-model files. Input evidence is also selective: only consumed feature rows and columns or referenced masks contribute. Configurations with missing, mismatched, or corrupt manifests are deleted before retraining; `fold_metrics.csv`, `predictions.csv`, and a manifest written last define completion, while plots and the aggregate comparison are derived and regenerated without retraining.

The pipeline validates globally unique configuration names, computes a visible execution plan before writes, supports a dry run, and permits only one execution per training output directory. Any stale configuration removes the aggregate comparison until every selected configuration has an Artefato de Avaliação Atual. Unselected configuration directories remain available for later reuse, while a renamed configuration has a new identity and must train again.

**Considered Options**: File existence was rejected because empty or stale CSVs appear complete. Hashing all project source or whole model-family files was rejected because adding an unrelated implementation would cause expensive retraining. Manual recipe versions were rejected because forgotten increments could silently reuse stale evaluations. Preserving stale results after a failed retrain was rejected in favor of fail-closed deletion.

**Consequences**: The first run after adoption retrains every configuration because existing artifacts have no verifiable manifest and shared training code changes. The provenance module and recipe registry become part of the test surface, and adding a model requires declaring its specific implementation, consumed inputs, and relevant dependencies.
